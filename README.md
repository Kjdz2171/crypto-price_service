### Crypto Price Service

Сервис для периодического получения цен с криптобиржи Deribit и предоставления внешнего API на FastAPI.

### Функциональность

- **Сбор цен с Deribit**: каждую минуту Celery‑задача запрашивает index price для `btc_usd` и `eth_usd` через HTTP API Deribit (`/public/get_index_price`) и сохраняет в PostgreSQL.
- **Хранение данных**: таблица `prices` содержит поля `id`, `ticker`, `price`, `timestamp` (UNIX timestamp в миллисекундах).
- **Внешнее API (FastAPI)**:
  - **GET `/prices`**: получить все сохранённые записи по тикеру (c необязательными параметрами `start_ts`, `end_ts` для фильтрации по дате).
  - **GET `/prices/latest`**: получить последнюю цену по тикеру.
  - **GET `/prices/by-date`**: получить записи по тикеру с фильтром по дате (`start_ts`, `end_ts`).
  - У всех методов обязателен query‑параметр `ticker`.

### Локальный запуск без Docker

1. **Установите зависимости**:

```bash
python -m venv .venv
.venv\Scripts\activate  # Windows
pip install -r requirements.txt
```

2. **Поднимите PostgreSQL** (локально или в Docker) и создайте базу:

- **DB_HOST**
- **DB_PORT**
- **DB_USER**
- **DB_PASSWORD**
- **DB_NAME**

Можно задать их через `.env` или переменные окружения.

3. **Запустите FastAPI**:

```bash
uvicorn app.main:app --reload
```

4. **Запустите Celery‑воркер с beat** (периодическая задача раз в минуту):

```bash
celery -A app.celery_app.celery_app worker -B --loglevel=info
```

### Запуск в Docker

Требуется Docker и Docker Compose.

```bash
docker compose up --build
```

Будут подняты сервисы:

- **app**: контейнер с FastAPI‑приложением (порт `8000`).
- **worker**: Celery‑воркер с beat, который раз в минуту опрашивает Deribit и пишет данные в PostgreSQL.
- **db**: PostgreSQL 16.
- **redis**: брокер и backend для Celery.

Документация Swagger будет доступна по адресу `http://localhost:8000/docs`.

### Примеры запросов

- **Все данные по тикеру**:

```bash
curl "http://localhost:8000/prices?ticker=btc_usd"
```

- **Последняя цена**:

```bash
curl "http://localhost:8000/prices/latest?ticker=btc_usd"
```

- **Фильтр по диапазону дат** (UNIX timestamp в мс):

```bash
curl "http://localhost:8000/prices/by-date?ticker=btc_usd&start_ts=1731456000000&end_ts=1731542400000"
```

### Архитектура (чистая архитектура / слои)

- **API (FastAPI)** — только обработка HTTP: валидация query-параметров, вызов сервиса, сериализация ответа. Зависимости получает через DI (`get_settings`, `get_db`, `get_price_service`).
- **Сервисный слой** — `PriceService`: оркестрирует операции «получить все по тикеру», «последняя цена», «по диапазону дат». Не знает про HTTP и БД-драйвер; зависит только от `PriceRepository`.
- **Репозиторий** — `PriceRepository`: единственная точка доступа к таблице `prices` (find_all_by_ticker, find_latest_by_ticker, add). Инкапсулирует SQL/SQLAlchemy.
- **Клиент биржи** — `DeribitClient`: один класс, отвечающий за общение с Deribit API (aiohttp). Настройки получает через `get_settings()`, без глобальных переменных.
- **Конфигурация и БД** — без модульных глобалов: настройки через `get_settings()` (кэш на процесс); в FastAPI движок и фабрика сессий создаются при старте и кладутся в `app.state`; воркер Celery использует процессовый кэш фабрики сессий только для инфраструктуры.

### Design decisions

- **FastAPI + async SQLAlchemy**: асинхронная обработка запросов и единая модель работы с PostgreSQL. `asyncpg` и асинхронные сессии дают предсказуемую производительность под нагрузкой.
- **Celery для периодических задач**: планирование раз в минуту через beat, масштабирование воркеров, привычный стек. Альтернатива — APScheduler внутри процесса — проще, но меньше возможностей по масштабированию и отказоустойчивости.
- **aiohttp как HTTP‑клиент для Deribit**: неблокирующие запросы, параллельный опрос нескольких индексов через `asyncio.gather`. Подходит под требование «применить aiohttp».
- **Redis как брокер Celery**: простой деплой, один контейнер, широко используется. Для продакшена с высокой надёжностью можно рассмотреть RabbitMQ.
- **UNIX timestamp в миллисекундах**: единый формат для хранения и фильтрации по времени; совместим с типичным форматом API бирж. В БД храним как есть, без конвертации в секунды.
- **Минимальная схема БД**: одна таблица `prices` (id, ticker, price, timestamp) покрывает задачу. При добавлении источников или типов цен схему можно расширить без смены слоёв.
- **Отсутствие глобальных переменных**: конфиг — через `get_settings()`; в веб-приложении движок и фабрика сессий живут в `app.state`; в воркере — один процессовый кэш фабрики сессий (инфраструктура), без глобального бизнес-состояния.

### Критерии оценки (соответствие заданию)

| Критерий | Реализация |
|----------|------------|
| **Чистая архитектура / чистый код** | Разделение на API → Service → Repository; один класс — одна ответственность; зависимости передаются явно (DI). |
| **Нейминг** | Единообразные имена: `ticker`, `find_all_by_ticker`, `get_latest`, `get_all_by_ticker`; суффиксы Dep для типов зависимостей. |
| **Отсутствие глобальных переменных** | Нет модульных глобалов для бизнес-логики; конфиг — `get_settings()`; БД в FastAPI — `app.state`; в воркере — один процессовый кэш фабрики сессий. |
| **ООП** | `DeribitClient`, `PriceRepository`, `PriceService` — классы с чёткой зоной ответственности; инжекция зависимостей через конструкторы. |
| **Понимание решений** | Секции «Архитектура» и «Design decisions» в README объясняют выбор технологий, слоёв и формат данных. |

### Тесты

- В каталоге `tests` есть базовые unit‑тесты для API:
  - Проверка поведения `/prices/latest`, когда данных нет.
  - Проверка получения последней цены после вставки записи в БД.

Запуск тестов:

```bash
pytest
# или
python -m pytest tests/ -v
```

**Windows:** если команда `python` не находится, попробуйте лаунчер `py -m pytest tests/ -v` или активируйте venv (`.\.venv\Scripts\Activate.ps1`) и затем `pytest tests/ -v`.

**Через Docker** (без установленного локально Python): при запущенных сервисах (`docker compose up -d` или в другом терминале) выполните:
```bash
docker compose run --rm app python -m pytest tests/ -v
```

Перед запуском тестов убедитесь, что переменные окружения для подключения к тестовой БД заданы (или используйте `.env`); в Docker они берутся из `docker-compose.yml`.
